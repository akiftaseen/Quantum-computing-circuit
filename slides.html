Problem and Motivation

Why this product exists and what gaps it closes

02 / 25

Pain Points in Existing Workflows

  • Backend dependence and paid compute tiers can block fast experimentation.
  • Beginner tools often hide internals; expert tools often overload UX.
  • Interoperability friction appears between visual builders and external stacks.
  • Offline and privacy-first use cases are underserved.

Product Response

  • Run fully in-browser with local persistence and no backend requirement.
  • Progressive disclosure: clean main interface plus deep Simulator Lab.
  • Expose both visual outputs and mathematical internals for learning and validation.
  • Ship OpenQASM-lite import/export, diagnostics, and round-trip checks.

Simulation Kernels

Math primitives and evolution operators

09 / 25

complex.ts

  • Complex arithmetic operations: add, sub, mul, conj, abs2, phase.
  • Formatting helper for stable, readable amplitude output.

gate.ts

  • Defines fixed and parametric gate matrices.
  • Covers 1Q, 2Q, 3Q gates and interaction families (XX, YY, ZZ).

simulator.ts

  • Applies single, controlled, 2Q, and 3Q gates via bitwise indexing.
  • Implements measurement collapse, partial trace, and Bloch extraction.

Why It Is Fast

The engine updates only affected amplitude pairs/triples instead of building giant dense matrices for each step.

Why It Is Testable

Pure, deterministic function boundaries allow logic suites to validate correctness without UI lifecycle coupling.

Noise and Density-Matrix Evolution

Realistic branch tracking for noisy execution paths

12 / 25

Implemented Noise Channels

  • amplitudeDamping
  • bitFlip, phaseFlip
  • readoutError correction model
  • timing controls: T1/T2, gate times, idle time

Calibration and Mitigation

  • Noise fit uses explicit grid: [0, 0.01, 0.02, 0.04, 0.06, 0.1].
  • Three nested loops evaluate 6^3 = 216 candidate combinations.
  • KL divergence score selects best-fit calibration tuple.
  • Readout mitigation inverts confusion matrix and renormalizes outputs.

Hardware Awareness and Routing

NISQ profile constraints integrated into design-time feedback

17 / 25

Built-In Hardware Profiles

  • Ideal All-to-All (reference baseline)
  • 5Q Line (NISQ)
  • 7Q Heavy-Hex Lite

Each profile includes basis-gate set, coupling edges, and noise-level metadata (T1/T2, readout, CX error).

Routing Strategy

  • Shortest-path traversal over coupling graph.
  • Auto-insertion of SWAP chains for nonadjacent interactions.
  • Mapping restore after routed operation to preserve logical indexing.
  • Returns overhead metrics, unroutable gate count, and compatibility score.
